Confirmed against the live API. Here's the accurate picture:

OpenAlex does not return full text content

The API gives you three text-related things, none of which is the paper body itself:

  1. Abstract — via abstract_inverted_index (already in your query). Reconstructable to plain text. This is the only actual text OpenAlex hands back.
  2. Full-text search — OpenAlex has indexed full text for many works and lets you search it (filter=fulltext.search:<term>), with a has_fulltext boolean and fulltext_origin (e.g. pdf, ngrams). But you can only query it — you cannot retrieve the indexed text.
  3. Links to open-access copies — the path to real full text.

To actually get full text: follow the OA links, then download/parse yourself

For open-access works, OpenAlex points you to a copy you fetch directly from the publisher/repository:

  • open_access.oa_url — best OA URL
  • primary_location.pdf_url / best_oa_location.pdf_url — direct PDF when available
  • locations[].pdf_url — all known hosting locations

From the live top-6:

Paper is_oa Direct PDF?
R manual ✓ gold ✅ cran.r-project.org/.../R-intro.pdf
Lowry (protein) ✓ hybrid ✅ jbc.org/.../pdf
ResNet ✓ green ⚠️ only a landing page (repositorio.unal.edu.co/handle/...), no direct PDF
Laemmli, Bradford ✗ closed ❌ no OA copy — full text not retrievable

So the reality: ~50–60% of works have an OA copy; only some of those give a direct PDF (the rest are landing pages you'd have to scrape), and closed-access papers can't be retrieved at all through OpenAlex. has_fulltext: True means OpenAlex indexed it for search — not that you can download it.

Options for you

  • Cheapest, already done: you're capturing abstract_inverted_index — good for most text/embedding/search use cases.
  • Add OA-link fields so you can fetch PDFs later: I'd add best_oa_location, locations, has_fulltext to the query (no extra cost, small size bump).
  • Full pipeline: a separate follow-on script that, for each OA record, downloads the pdf_url and extracts text (e.g. pypdf/pdfminid) — bounded by OA availability and much slower (10k HTTP downloads + parsing, plus polite crawling of many publisher hosts). Doable but a meaningfully bigger job.
