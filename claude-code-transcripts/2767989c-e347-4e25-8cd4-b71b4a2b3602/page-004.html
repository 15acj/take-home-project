Since shards are keyed by citation rank (not graph locality), consecutive clicks usually hit different shards, so I'd actually lean 100 papers/shard (100 shards, ~47 KB each) for the snappiest click response — CDNs handle many small immutable objects fine.

Compression — one correction to the plan

The plan said "Brotli q11 + gzip," but I need to flag: Python's stdlib has gzip but not brotli (just confirmed — all my measurements above are gzip). So to actually produce Brotli at build time, we'd need one of: - Node's built-in zlib.brotliCompressSync — zero extra install, and the layout step already uses Node, so this is the clean choice; - pip install brotli, or the brotli CLI; - or let the CDN Brotli-compress on the fly (Cloudflare/Fastly/CloudFront all do), and just ship gzip as the static fallback.

Brotli q11 runs ~15–20% smaller than gzip on this text, so a 250-shard goes ~115 KB → ~95 KB (a 100-shard ~47 → ~40 KB).
