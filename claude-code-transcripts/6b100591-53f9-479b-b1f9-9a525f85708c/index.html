#82026-07-11T18:59:09.416Z

do we use brotli or gzip encoding when displaying the graph?

5 bash · 1 toolsearch · 1 fetch

On the wire it's a mix — and one part of the first paint isn't compressed at all:

JSON artifacts (manifest.json, nodes-text.json, search-kw.json, detail shards): the client fetches the raw paths and the serving layer negotiates. On Vercel, application/json is on the compression allowlist and Brotli takes precedence over gzip whenever the browser advertises it (all modern browsers do), so in production these arrive Brotli-encoded (e.g. nodes-text.json: 1.13 MB raw → ~316 KB br). On local next start it's gzip only (~431 KB) — Next's built-in compression doesn't do Brotli.

Binary artifacts (nodes.bin, edges.bin, positions.bin): neither — they're served uncompressed. I verified both ends: locally they come back with no Content-Encoding (105 KB / 96 KB / 57 KB raw on the wire), and Vercel's docs confirm application/octet-stream is not on their compression allowlist, so the edge won't compress them either.

The practical impact is modest: positions.bin is quantized noise (compression saves ~1%), but nodes.bin + edges.bin would shrink from ~200 KB to ~105 KB brotli'd. The .gz/.br sidecars we build are currently unused by the app — they only matter for a CDN setup where you serve them explicitly.

If you want the last ~100 KB: the clean fix on Vercel is rewrites in next.config.js mapping /data/v1/*.bin → the .br sidecar with a Content-Encoding: br header (modern browsers all accept it). Happy to wire that up if you'd like.
