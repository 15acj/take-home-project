# Copilot tool: set graph filters from natural language

## Context

The Copilot chat (streaming Next API route `web/pages/api/copilot.ts` + the `send`
action in `web/components/CitationAtlas.tsx`) currently only answers questions
about selected papers. Add a **tool** the model can call to translate
natural-language requests into the app's existing filter state, so users can say
things like:

- "show me the machine learning papers from 2025 that have pdf links"
- "filter to papers with more than 5000 citations"
- "reset the filters"
- "show me articles from 2020 to 2026 in genetics and neuroscience"

…and the left FilterPanel + 3D graph update live. Commands are **incremental**
(adjust only the dimensions the user mentions; keep the rest) with an explicit
**reset** — the "reset the filters" example implies exactly this.

Model stays `claude-haiku-4-5` (supports tool use).

## How filters already work (reuse, don't reinvent)

- Filter state is in the zustand store (`web/lib/store.ts`): `keyword` /
  `keywordApplied`, `topN`, `yearMin`, `yearMax`, `minCites`,
  `activeFields: Record<ClusterKey,bool>`, `activeTypes: Record<ContentTypeKey,bool>`,
  `availOA`, `availPdf`, `availGrobid`. Constants `YEAR_MIN=1935`, `YEAR_MAX=2026`,
  `CITE_MAX=40000`, `TOPN` = `[100,1000,5000,10000]`. `filterDefaults()` resets all.
- The `applyFilters` effect (`CitationAtlas.tsx:144-161`) subscribes to those keys
  and recomputes the visible graph via `computeFilter` (`web/lib/filter.ts`)
  **automatically** — writing the store keys is all that's needed.
- Field clusters (`web/lib/fieldClusters.ts`): `genetics` (Genetics & Molecular
  Biology), `ai` (Machine Learning & AI), `physics`, `chemistry`, `medicine`,
  `neuro` (Neuroscience), `cs` (Computer Science & Math), `economics`. Content
  types (`web/lib/contentTypes.ts`): `article`, `book`, `review`, `other`.
- FilterPanel's own mutations (`web/components/FilterPanel.tsx`) are the pattern to
  mirror: fields via `Object.fromEntries(CLUSTER_KEYS.map(k => [k, on]))`, keyword
  via setting `keyword` + `keywordApplied`, etc.

## Approach

### Stream protocol → NDJSON
Today the route streams raw text and the client appends it. To carry both text
and structured filter actions, stream **newline-delimited JSON**, one object/line:
- `{"t":"text","v":"<delta>"}` — assistant text (streamed into the bubble as now)
- `{"t":"action","name":"set_filters"|"reset_filters","input":{...}}` — filter action

### Route — `web/pages/api/copilot.ts`
- Add two tools to the `client.messages.stream(...)` call (`tool_choice` auto):
  - **`set_filters`** — all optional: `fields` (enum[] of the 8 cluster keys —
    replaces the shown set), `content_types` (enum[] — replaces), `year_min` /
    `year_max` (int), `min_citations` (int), `keyword` (string; `""` clears),
    `require_pdf` / `require_open_access` / `require_full_text` (bool), `top_n`
    (enum 100/1000/5000/10000). Describe each cluster **by label** so the model
    maps NL→key; state that only-provided keys change and `fields`/`content_types`
    **replace** the active set.
  - **`reset_filters`** — no params.
- Stream text deltas as `{"t":"text",...}` lines (`stream.on("text")`). After
  `stream.finalMessage()`, emit one `{"t":"action",...}` line per `tool_use` block
  with the parsed input. **Single request, no tool_result round-trip** — the
  model's one-sentence preamble (instructed) is the confirmation.
- System prompt additions (keep existing science/research scope + paper
  grounding): use the tools only when the user asks to change/reset what the graph
  shows; **write one concise sentence confirming the change, then call the tool**;
  prefer field clusters for broad domains and `keyword` for specific topics/terms;
  apply only the dimensions the user mentions (leave the rest); to add/remove
  fields or types, send the **full new set** (current filter state is provided);
  constraints (year 1935–2026, citations 0–40000, top_n set). Inject a
  **current-filters summary** built from a new `filters` field in the body.
- Body becomes `{ question, papers, history, filters }`.
- Error handling: pre-stream error → 500 JSON (client shows error bubble);
  mid-stream error → emit a `{"t":"text","v":"…error…"}` line and end.

### Client — `web/components/CitationAtlas.tsx`
- In `send`, add the current `filters` snapshot to the POST body from
  `S.getState()`: on-field keys, on-type keys, `year_min/max`, `min_citations`,
  `keyword`, `require_pdf/open_access/full_text`, `top_n`.
- Rewrite the stream reader to parse **NDJSON** (buffer partial lines; `JSON.parse`
  each complete line):
  - `text` → accumulate into the assistant bubble (keep the sticky-autoscroll and
    "append on first content, clear typing" behavior already there).
  - `action` → call a new `applyFilterAction(name, input)`.
  - If a turn produced an action but no text, show a short templated confirmation.
- Add `applyFilterAction(name, input)` (component scope):
  - `reset_filters` → `S.setState(filterDefaults())`.
  - `set_filters` → build a `Partial<AtlasState>` and `S.setState`:

### what is the difference between precision and recall?
Precision and recall are two fundamental metrics for evaluating classifier performance, especially in imbalanced or high-stakes classification tasks:

**Precision** is the fraction of positive predictions that were actually correct: TP / (TP + FP). It answers "of the cases I predicted as positive, how many were right?" High precision means few false positives—useful when the cost of incorrectly flagging something is high (e.g., medical diagnoses, spam detection).

**Recall** is the fraction of actual positives the model caught: TP / (TP + FN). It answers "of all the truly positive cases, how many did I find?" High recall means few false negatives—useful when missing a positive case is costly (e.g., disease screening, security threat detection).

The classic tradeoff: improving one often worsens the other. A classifier that predicts "positive" for everything has perfect recall (catches all true positives) but terrible precision (many false alarms). One that almost never predicts positive has high precision but poor recall.

In practice, you choose based on your problem: precision matters more in fraud detection (false positives are expensive), while recall matters more in cancer screening (false negatives are dangerous). The **F1 score** is their harmonic mean when you want to balance both.

These metrics appear across machine learning research—detection tasks, information retrieval, medical AI, and many classification papers in the graph. If you'd like to see how they're used in specific research contexts, you can click papers in the graph or filter to a particular domain.

"NDJSON reader" is the client-side loop in web/components/CitationAtlas.tsx (inside the send action) that reads the copilot's HTTP response stream and parses it. Here's the what and why.

NDJSON = newline-delimited JSON

A streaming format where each line is one complete JSON object:

{"t":"text","v":"I'll filter the graph "}
{"t":"text","v":"to machine learning papers."}
{"t":"action","name":"set_filters","input":{"fields":["ai"]}}

I switched the copilot stream to this because the reply now carries two kinds of things — assistant text and structured filter actions. Plain text couldn't distinguish them; tagging each line with a t field can. The route writes these lines (writeLine(...) in copilot.ts); the reader consumes them.

What the reader does

fetch gives res.body, a stream of raw byte chunks. Those chunks don't align to line or JSON boundaries — one network chunk might be half a line, or two-and-a-half lines. So the reader buffers and splits on newlines:

const reader = res.body.getReader();
const decoder = new TextDecoder();
let buf = "";

for (;;) {
  const { done, value } = await reader.read();   // next byte chunk
  if (done) break;
  buf += decoder.decode(value, { stream: true }); // bytes → text, incrementally
  let nl;
  while ((nl = buf.indexOf("\n")) >= 0) {          // for each complete line…
    handleLine(buf.slice(0, nl));                  //   parse + dispatch it
    buf = buf.slice(nl + 1);                       //   keep the remainder
  }
}
buf += decoder.decode();          // flush decoder
if (buf.trim()) handleLine(buf);  // a trailing line with no final newline

handleLine is the dispatcher:

const obj = JSON.parse(line);
if (obj.t === "text")   pushText(obj.v);                     // stream into the chat bubble
else if (obj.t === "action") applyFilterAction(obj.name, obj.input); // change the filters

Two details that matter: - TextDecoder({stream:true}) — decodes bytes to text across chunk boundaries, so a multi-byte UTF-8 character split between two chunks isn't corrupted. - The while loop per chunk — one chunk can contain several lines, so it drains all complete lines and leaves any partial line in buf for the next chunk. The try/catch in handleLine also means a not-yet-complete or malformed line is skipped rather than throwing.

vs. before

Previously the stream was plain text and the reader just appended every chunk straight to the assistant message. Now it's a framed protocol: text lines still stream into the bubble (with the same sticky-autoscroll), and action lines drive the graph filters — which is what let the same streamed reply both say "filtering to ML papers…" and actually apply the filter.

# Turbopuffer TypeScript/JavaScript SDK Query Guide

Based on the documentation, here's how to use the TypeScript/JavaScript SDK:

## Client Setup

```typescript
import Turbopuffer from 'turbopuffer';

const tpuf = new Turbopuffer({
  apiKey: process.env.TURBOPUFFER_API_KEY,
  region: 'gcp-us-central1'
});

const ns = tpuf.namespace('my-namespace');
```

## Vector Search (ANN)

```typescript
const result = await ns.query({
  rank_by: ["vector", "ANN", [0.1, 0.2, 0.3]],
  limit: 10,
  include_attributes: ['title', 'content']
});

console.log(result.rows);
// Returns: [{ id: 1, $dist: 0.15, title: '...', content: '...' }, ...]
```

## Full-Text Search (BM25)

```typescript
const result = await ns.query({
  rank_by: ['content', 'BM25', 'quick fox'],
  limit: 10,
  include_attributes: ['title', 'content']
});
```

```typescript
const response = await ns.multi_query({
  queries: [
    { rank_by: ["vector", "ANN", [1.0, 0.0]], limit: 10 },
    { rank_by: ["attr1", "BM25", "quick fox"], limit: 10 }
  ],
  rerank_by: ["RRF"]
});

console.log(response.results);
// Returns combined ranked results with "$dist" scores
```

**Key response fields:** `rows` (array of documents), `$dist` (ranking score), `id` (always included)

# Copilot tool: find similar papers (hybrid semantic search)

## Context

The copilot can answer questions and set graph filters. Add a **`find_similar_papers`
tool**: given a query (a topic the user describes, or "papers similar to this
one" synthesized from the selected paper), run **hybrid search** (dense vector +
BM25) over the paper embeddings already in **turbopuffer**, and return the most
relevant papers. The results render in the chat as an **interactive card** — each
paper has a checkbox (checked by default) and there's a button to **add the
checked papers to the graph selection** (so they highlight in the 3D atlas and
enable follow-ups like "compare these").

Retrieval must: embed the query with **OpenAI**, **expand the query with Haiku**
(a couple rephrasings) to widen recall, fuse dense+BM25 across all variants, apply
a **relevance threshold** to drop weak matches, and return up to **5–10** results
ranked by relevancy.

## Infrastructure already in place (reuse — see `scripts/embed_turbopuffer.py`)

- Embeddings: **OpenAI `text-embedding-3-small` (1536-d)** over `title\n\nabstract`.
  The query MUST use this exact model to match the stored vectors.
- Turbopuffer namespace: `cosine_distance` metric, **full-text search enabled on
  the `text` attribute** (hybrid ready). Row schema: `id` (OpenAlex short id),
  `vector`, `text`, **`rank` (int) == the graph node index** (the join key back to
  the app's selection), `title`, `year`, `cited_by_count`, `field`, `domain`, `doi`.
- Creds already in `scripts/.env`: `OPENAI_API_KEY`, `TURBOPUFFER_API_KEY`,
  `TURBOPUFFER_REGION`, `TURBOPUFFER_NAMESPACE`.
- TS SDKs (confirmed): `turbopuffer` — `new Turbopuffer({ apiKey, region })` →
  `tpuf.namespace(ns).query({ rank_by, limit, include_attributes })` → `res.rows`

  `client.embeddings.create({ model, input })`.

// Server-only. Hybrid (dense vector + BM25) semantic search over the paper
// embeddings in turbopuffer, for the copilot's find_similar_papers tool.
//
// Pipeline: expand the query with Haiku (a couple rephrasings) → embed all
// variants with OpenAI (text-embedding-3-small, matching the ingested vectors)
// → per variant run a vector ANN query and a BM25 query in turbopuffer → fuse
// with Reciprocal Rank Fusion in TS → keep only results whose best cosine
// similarity clears a threshold (drops irrelevant papers) → return the top N
// ranked by fused score.
import type Anthropic from "@anthropic-ai/sdk";
import OpenAI from "openai";
import { Turbopuffer } from "@turbopuffer/turbopuffer";
import type { SimilarResult } from "./store";

const EMBED_MODEL = "text-embedding-3-small"; // MUST match the corpus embeddings
const EXPAND_MODEL = "claude-haiku-4-5";
const RRF_K = 60;
const CANDIDATES = 40; // top_k per sub-query
// Min cosine similarity (1 - cosine_distance) for a paper to count as relevant.
// text-embedding-3-small puts related papers ~0.35–0.6 and unrelated ones <~0.25.
const SIM_THRESHOLD = 0.35;
const ATTRS = ["rank", "title", "year", "cited_by_count", "field", "doi"];

// turbopuffer rows carry the included attributes dynamically alongside id/$dist.
interface Row {
  id: string | number;
  $dist?: number;
  rank?: number;
  title?: string;
  year?: number;
  cited_by_count?: number;
  field?: string;
  doi?: string;
}

export function similarSearchConfigured(): boolean {
  return Boolean(
    process.env.OPENAI_API_KEY &&
      process.env.TURBOPUFFER_API_KEY &&
      process.env.TURBOPUFFER_REGION &&

}

// Ask Haiku for a couple of alternative phrasings to widen recall. Best-effort:
// any failure (or unparseable output) just yields no extra variants.
async function expandQuery(anthropic: Anthropic, query: string): Promise<string[]> {
  try {
    const msg = await anthropic.messages.create({
      model: EXPAND_MODEL,
      max_tokens: 200,
      system:
        "You rephrase a research-paper search query to widen semantic-search recall. Return ONLY a compact JSON array of exactly 2 alternative phrasings (different wording and synonyms, same intent, suitable for searching academic paper titles and abstracts). No prose, no markdown, no code fences.",
      messages: [{ role: "user", content: query }],
    });
    const text = msg.content.map((b) => (b.type === "text" ? b.text : "")).join("");
    const start = text.indexOf("[");
    const end = text.lastIndexOf("]");
    if (start === -1 || end <= start) return [];
    const arr = JSON.parse(text.slice(start, end + 1));
    return Array.isArray(arr)
      ? arr.filter((s): s is string => typeof s === "string" && s.trim().length > 0)
      : [];
  } catch {
    return [];
  }
}

export async function findSimilarPapers(opts: {
  anthropic: Anthropic;
  query: string;
  limit: number;
  excludeRanks: number[];
}): Promise<SimilarResult[]> {
  const query = opts.query.trim();
  if (!query) return [];
  const limit = Math.max(1, Math.min(10, Math.round(opts.limit || 8)));

  const openaiKey = process.env.OPENAI_API_KEY!;
  const region = process.env.TURBOPUFFER_REGION!;
  const namespace = process.env.TURBOPUFFER_NAMESPACE!;

  // 1. Expand → variant set (original + up to 2 rephrasings).
  const rephrasings = await expandQuery(opts.anthropic, query);
  const variants = Array.from(
    new Set([query, ...rephrasings].map((s) => s.trim()).filter(Boolean)),
  ).slice(0, 3);

  // 2. Embed all variants in one call.
  const openai = new OpenAI({ apiKey: openaiKey });
  const emb = await openai.embeddings.create({ model: EMBED_MODEL, input: variants });
  const vectors = emb.data
    .slice()
    .sort((a, b) => a.index - b.index)
    .map((d) => d.embedding as number[]);

  // 3. Hybrid retrieve: vector ANN + BM25 per variant, all in parallel.
  const tpuf = new Turbopuffer({ apiKey: process.env.TURBOPUFFER_API_KEY!, region });
  const ns = tpuf.namespace(namespace);
  const annP = variants.map((_, i) =>
    ns.query({
      rank_by: ["vector", "ANN", vectors[i]] as [string, "ANN", number[]],
      top_k: CANDIDATES,
      include_attributes: ATTRS,
    }),
  );
  const bm25P = variants.map((v) =>
    ns.query({
      rank_by: ["text", "BM25", v] as [string, "BM25", string],
      top_k: CANDIDATES,
      include_attributes: ATTRS,
    }),
  );
  const [annRes, bm25Res] = await Promise.all([Promise.all(annP), Promise.all(bm25P)]);

  // 4. RRF fuse across every list; track best cosine similarity from ANN lists.
  interface Acc { score: number; bestSim: number; row: Row; }
  const byId = new Map<string, Acc>();
  const addList = (rows: Row[], isAnn: boolean) => {
    rows.forEach((row, idx) => {
      const id = String(row.id);
      let acc = byId.get(id);
      if (!acc) { acc = { score: 0, bestSim: -1, row }; byId.set(id, acc); }
      acc.score += 1 / (RRF_K + idx);
      if (isAnn && typeof row.$dist === "number") acc.bestSim = Math.max(acc.bestSim, 1 - row.$dist);
      if (!acc.row.title && row.title) acc.row = row; // prefer a row with attributes
    });
  };
  annRes.forEach((r) => addList(r.rows as unknown as Row[], true));
  bm25Res.forEach((r) => addList(r.rows as unknown as Row[], false));

  // 5. Threshold on the dense signal (drops irrelevant papers and BM25-only
  //    keyword hits), exclude the papers already selected, rank by fused score.
  const exclude = new Set(opts.excludeRanks);
  return [...byId.values()]
    .filter((a) => a.bestSim >= SIM_THRESHOLD && a.row.rank != null && !exclude.has(Number(a.row.rank)))
    .sort((a, b) => b.score - a.score)
    .slice(0, limit)
    .map((a) => ({
      rank: Number(a.row.rank),
      id: String(a.row.id),
      title: String(a.row.title ?? ""),
      year: Number(a.row.year ?? 0),
      cited_by_count: Number(a.row.cited_by_count ?? 0),
      field: String(a.row.field ?? ""),
      doi: a.row.doi ? String(a.row.doi) : null,
      similarity: Math.round(a.bestSim * 100) / 100,
    }));
}

If you'd prefer the list sorted strictly by % match instead, that's a one-line change.
