Yes — the hybrid mode already uses one: Reciprocal Rank Fusion (RRF). It's the rerank_by=("RRF",) argument in the multi_query call (cmd_query, the hybrid branch).

How RRF works

Each sub-query (vector ANN and BM25) returns its own ranked list. RRF ignores the raw scores — which live on incomparable scales (cosine distance vs. BM25 term-frequency) — and fuses purely on rank position. Each document gets:

$$\text{score}(d) = \sum_{q \in \text{queries}} \frac{1}{k + \text{rank}_q(d)}$$

where rank_q(d) is the document's 1-based position in query q's list, and k is a smoothing constant (turbopuffer defaults to k = 60, the value from the original paper). Documents are then sorted by summed score. A doc that ranks #1 in both lists scores highest; a doc appearing in only one list still contributes but is outweighed by anything ranking well in both.

Why rank-based instead of score-based: it sidesteps the normalization problem entirely. You don't have to decide how to scale a cosine distance against a BM25 score or pick a blend weight — position is the common currency. This is exactly why Attention Is All You Need topped every mode in the earlier test: it ranked #1 in both the vector and BM25 lists, so RRF put it #1 too.

Tuning it

turbopuffer lets you pass RrfParams to adjust it — the main knob is k. Lower k sharpens the emphasis on top-ranked hits; higher k flattens the contribution across the list. If you wanted to expose that, it'd be a one-line change in the rerank_by tuple:

rerank_by=("RRF", {"k": 40}),

Alternatives (if you outgrow RRF)

  • Weighted score fusion — normalize each list's scores (min-max or z-score) and take a weighted sum. More control (you can favor semantic over lexical), but you own the normalization and weight-tuning.
  • A learned re-ranker / cross-encoder (e.g. a Cohere Rerank or a local cross-encoder model) applied to the fused top-N. This is the usual next step for real quality gains: RRF cheaply narrows 9,875 → ~50 candidates, then the cross-encoder re-scores those 50 by jointly reading query + document. turbopuffer's own docs recommend exactly this two-stage pattern.

For 9,875 papers, RRF alone is a well-calibrated default — I'd only reach for a cross-encoder if you find the top few results aren't ordered the way you want.

Yes. The embedding input is title + reconstructed abstract, built in build_embed_text (embed_turbopuffer.py:99-104):

def build_embed_text(work: dict) -> str:
    """title + reconstructed abstract (title-only when no abstract), length-capped."""
    title = (work.get("title") or "").strip()
    abstract = reconstruct_abstract(work.get("abstract_inverted_index")) or ""
    text = f"{title}\n\n{abstract}".strip() if abstract else title
    return text[:MAX_EMBED_CHARS]

Specifics: - Format: title + blank line + abstract, as one string per paper. - Abstract source: reconstructed from OpenAlex's abstract_inverted_index (word→positions) via reconstruct_abstract — the raw JSONL doesn't store plain-text abstracts. - No-abstract papers: fall back to title only (some records have no abstract_inverted_index). - Length cap: truncated to MAX_EMBED_CHARS = 8000 * 4 (~8k tokens) to stay under OpenAI's 8191-token input limit. In practice abstracts are short enough that this rarely triggers.

The same combined string is also stored as the text attribute in turbopuffer and indexed for BM25 (cmd_ingest), so both the vector and keyword paths search over identical title+abstract text.

One thing worth flagging: I embed title + abstract but not authors, venue, year, or topics/keywords. Those are stored as filterable attributes (field, domain, year, cited_by_count) but don't influence semantic similarity. If you wanted, say, a query like "2015 deep learning papers" to rank by recency, that'd be a filter on year rather than something the embedding captures.
